comprehensive school management system inspired by the functionalities of
IBEAMS. The system is designed to manage all aspects of a school's
administrative and academic operations, including student enrollment, class
management, attendance, fees, examinations, reporting, and parent/teacher
communication.

2. Core Modules & Features
The system will be organized into the following core modules:
· Administration/Setup: Academic Year, School Profile, User Management,
System Settings.
· Student Management: Add/Edit/Import Students, Student Status, Siblings,
Contacts (Parents/Guardians).
· Academic Management: Class Management, Teacher Assignment, Timetable
Scheduling.
· Attendance: Daily & Monthly Registers, Reporting, SMS Notifications for
Absences.
· Examinations: Official Exams & Progressive Tests, Results Entry, Grading.
· Reporting: End of Term/Year Reports (Performance, Behavior, Attendance),
Certificates.
· Communication: Teacher & Parent Portals, SMS Integration, Email
Notifications.
· Student Diaries: Homework, Syllabus Tracking, Hifz/Revision Logs.
· Merits & Store: Points System, Achievements, Product Store with
Merits/Currency and certificates generating system for end of semester.
· Fees & Finance: Fees Structure Creation, Invoice Generation, Payment
Tracking.

3.2 Database Management Information
· Database Server: For the Windows application, a central SQL server (e.g.,
PostgreSQL, MySQL) is required. The mobile app will act as a client to this
server.
· API: A secure RESTful API will act as the intermediary between the
Windows/Android applications and the database. All data interactions will be
through API calls.
· Authentication: JWT (JSON Web Tokens) will be used for secure user
authentication and authorization.
· Migrations: Database migrations (e.g., using Alembic for Python, Laravel's
Migrations, or Flyway) should be used to version-control the database schema
and make updates seamless.
· Offline Capability: For Android, implement a local database (e.g.,
SQLite/Room) that can synchronize with the central server when an internet
connection is available. This is crucial for teachers taking registers.
4. Technical Architecture & Stack Recommendations
4.1 General
· Architecture: Client-Server (RESTful API). Multiple clients (Windows Desktop
App, Android App, Future Web Portal) communicate with a centralized API
server.

=== PAGE 7 ===
· API Framework: Node.js (with Express.js), Python (with Django REST
Framework or FastAPI), or C# (.NET Core). Choose based on the team's
expertise.
· Database: PostgreSQL (robust, feature-rich) or MySQL.
4.2 Windows Application
· Framework: .NET MAUI (Multi-platform App UI) or WPF (Windows
Presentation Foundation). MAUI allows for a single codebase to target Windows,
Android, and iOS, but for pure Windows WPF is a very mature and stable choice.
· UI Design: XAML for a clean, responsive, and data-driven interface.
4.3 Android Application
· Native: Kotlin with Android SDK. Provides the best performance and access to
device features (SMS, Camera).
· Cross-Platform: Flutter (Dart) or React Native (JavaScript). Both allow for a
single codebase for Android and iOS (if needed). Flutter is recommended for its
performance and UI consistency.
· Offline Storage: Room Persistence Library (Kotlin) or Hive (Flutter).
4.4 Hosting & Infrastructure
· Server: Cloud-based (AWS, Google Cloud, Azure) or an on-premise server.
· Web Server: Nginx or Apache.
· SSL/TLS: Essential for encrypting data in transit and securing the API.
5. Detailed Module Implementation Notes
5.1 Setup & Configuration
· Academic Year: The system must be built around the concept of an academic
year. All core entities (students, classes, fees) are linked to a year. A "New
Academic Year Wizard" is essential to migrate data (classes, students, fee
structures) without losing historical records.
· School Profile: Easily customizable settings (logo, public address,
financial/sender emails) that are used globally in PDFs and emails.

=== PAGE 8 ===
5.2 People Management (Students, Teachers, Contacts)
· Students & Contacts (Parents): Implement a many-to-many relationship. A
contact can be associated with multiple students (siblings). Contact emails must
be unique for parent portal login.
· Import/Export: Robust CSV import and export for all entities.
· Student Status: The student lifecycle (waiting_list -> accepted -> current -> left)
must be managed. Status changes should trigger actions (e.g., "Student Left"
wizard to delete invoices/register entries).
5.3 Class Management & Timetables
· Classes & Teachers: A class has one "Form Teacher" but can have many
additional teachers with granular permissions (register, reports, homework, etc.).
· Timetable Templates: Allow creating reusable timetable templates (e.g.,
Morning, Evening, Ramadhan) and applying them to multiple classes.
· Classroom Management: A separate feature to add and manage
classrooms/buildings and assign them to timetable slots.
5.4 Attendance (Registers)
· Daily vs. Monthly: Design the register system to support both. Daily is for
day-to-day tracking; Monthly is a summary for reporting.
· Data Flow: Daily register data should be able to auto-fill monthly registers.
· Teacher Experience: The teacher's workflow should be streamlined. They should
see their assigned timetable for the day. By default, all students are marked
"Present." The teacher toggles to "Absent" or "Late."
· Offline First: The Android app must function fully offline. All register data is
saved locally and synced when a connection is available.
5.5 Fees Management
· Fee Structure: Define fee items (Tuition, Transportation) per class or per
student. Invoices are generated based on this structure for all students.
· Invoicing & Payments: Generate invoices, track payments, and handle partial
payments. Creating separate invoices for class-specific fees (Independent Fees) is
a required feature.

=== PAGE 9 ===
5.6 Examinations & Reporting
· Exams vs. Progressive Tests: Both have the same structure but serve different
purposes. Official exam results are used in formal end-of-term/year reports;
progressive test results are for ongoing tracking.
· End of Term/Year Report: A sophisticated system that pulls data from registers
(attendance), exams (results), and manual teacher input (behavior, effort,
progress). The manual scores (1-5) are converted to descriptive sentences via
configurable "Paragraph Templates."
5.7 Communication & Portals
· Portals (Teacher/Parent/Student): The system does not have a built-in public
portal. Instead, it provides "Portal Files" (PHP files) that an institution can host
on their own website. These files act as a proxy, securely communicating with the
API. This is a unique design choice that prioritizes security and branding for the
school.
· SMS Integration: Supports two methods:
1. Via Android App: Uses the device's SIM card to send SMS. This is a
cost-effective method but requires the app to be running on a physical device
with an active SIM.
2. Via Providers: Integrate with SMS gateway APIs like ClickSend or
VoodooSMS.
6. Key Features & Implementation Considerations
· Security: The system URL should be kept private. Portal files are used to keep
this URL hidden and ensure secure communication.
· Data Migration:
· Provide tools for each step of the academic year setup (migrate classes,
students, fee structures).
· Allow importing data via CSV.
· Customization:
· Custom Fields for students and other entities.
· Customizable Date/Time formats.
· Age Group customization.

=== PAGE 10 ===
· Scalability: The application should be designed to handle a large volume of
users, data, and concurrent requests. Use caching, database indexing, and
efficient queries.
· Notifications: Email notifications for status changes, fee reminders, and new
reports. SMS notifications for absences.
· Reporting: Generate dynamic PDF reports (Student Reports, Class Lists,
Financial Reports) using a library like iText, PDFBox, or wkhtmltopdf.

8. Glossary
· PK: Primary Key (Unique identifier for a row in a database table).
· FK: Foreign Key (A field in one table that refers to the Primary Key of another
table).
· RESTful API: An application programming interface that uses HTTP requests to
GET, POST, PUT, and DELETE data.
· JWT: JSON Web Token, a compact and self-contained way for securely
transmitting information between parties as a JSON object. Used for
authentication.
· CSV: Comma-Separated Values, a simple text format for storing tabular data.
